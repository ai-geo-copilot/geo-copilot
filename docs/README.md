# GEO 项目文档入口

状态：active  
最后更新：2026-06-28

本文件是 `docs/` 的唯一正式入口。它不记录当前开发完成度，也不裁定实现真相；当前实现、优先级、验证结果、阻塞和下一步一律以 `DEVELOPMENT_STATUS.md` 为准。

## 1. 使用方式

开始任何编码、调试、评审、架构或文档工作前：

1. 先读 `DEVELOPMENT_STATUS.md`。
2. 再按本文件找到对应的正式设计文档。
3. 只在需要时进入 `模块开发补充/`。

不要把聊天记录、根目录散落文档或 `开发过程中定义文件/` 当作正式事实源。

## 2. 正式文档分层

### 2.1 永久主文档

这些文档描述长期稳定的产品定义、架构边界和协作规则，应保持低频、克制、可长期复用：

| 文档 | 角色 | 适合回答的问题 |
|---|---|---|
| `DEVELOPMENT_STATUS.md` | 当前实现与验证事实源 | 现在做到哪一步、验证过什么、当前优先级是什么 |
| `GEO项目总纲.md` | 产品宪法与设计不变量 | 这个产品到底是什么、不是什么、长期不能破坏什么 |
| `GEO实施路线与架构决策.md` | 规范化系统蓝图 | 系统如何分层、数据谁拥有、工作流怎么拆 |
| `GEO架构技术栈与工具整合建议.md` | 技术选型门禁 | 该不该引入某个框架、库、队列、向量库或外部平台 |
| `GEO五人团队分工协作与验收标准.md` | 长期协作规则 | 一个变更应该由谁主导、如何验收、如何更新文档和状态 |
| `GEO论文优化方法知识库.md` | 方法来源与研究基线 | 方法卡依据来自哪里、研究资料如何沉淀 |

### 2.2 模块补充文档

这些文档服务于某个模块或阶段性设计，属于“主文档下的专门方案”：

| 文档 | 角色 |
|---|---|
| `模块开发补充/HTTP层GEO开发流程与完成标准.md` | Page Evidence / HTTP 主链路补充方案 |
| `模块开发补充/知识库架构技术开发方案.md` | Research KB / Method Pack / Selector / Strategy 补充方案 |
| `模块开发补充/DeepSeek诊断层模型调用边界开发方案.md` | Diagnosis 模型边界补充方案 |
| `模块开发补充/Conversation与GEOCopilotChat层开发方案.md` | Copilot Conversation 补充方案 |
| `模块开发补充/前端页面与接口对接开发方案.md` | 前端 Workbench / Report 接口与页面补充方案 |
| `模块开发补充/PageEvidence启发式规则质量优化方案.md` | Page type / heuristic / fixture / 评估补充方案 |
| `模块开发补充/商业产品化重构与Agent架构方案.md` | 长期产品化、工作流和 Agent 化补充方案 |

### 2.3 归档与讨论文档

以下目录不参与正式裁决：

- `开发过程中定义文件/`
- `后期开发/`

它们可以保留调研、临时设计和 backlog，但不能替代正式主文档。

## 4. 文档边界

为了让文档可长期维护，必须保持清晰边界：

- `DEVELOPMENT_STATUS.md` 只写已验证事实、当前优先级、验证命令、阻塞和下一步。
- `GEO项目总纲.md` 只写产品定义、设计原则、长期不变量和非目标。
- `GEO实施路线与架构决策.md` 只写系统蓝图、上下文边界、状态所有权和演进路线。
- `GEO架构技术栈与工具整合建议.md` 只写技术选择、引入门禁、替换点和反模式。
- 模块补充文档只写该模块的专项设计，不重复顶层状态。

如果一份正式文档开始大量重复“当前已完成什么”，应把这些内容收回 `DEVELOPMENT_STATUS.md`。

## 5. 冲突裁决

- 当前实现状态、已验证结果、当前优先级、阻塞、下一步：以 `DEVELOPMENT_STATUS.md` 为准。
- 产品定义、承诺边界、长期不变量：以 `GEO项目总纲.md` 为准。
- 系统分层、状态所有权、工作流和演进路线：以 `GEO实施路线与架构决策.md` 为准。
- 技术采用或拒绝条件：以 `GEO架构技术栈与工具整合建议.md` 为准。
- 模块级专项方案：在不与以上文档冲突时成立；若冲突，以上位文档为准。

## 6. 更新规则

发生以下变更时，必须同步更新对应文档和 `DEVELOPMENT_STATUS.md`：

- 公开 API contract 变化
- 核心领域对象变化
- 数据所有权变化
- 工作流拆分变化
- 技术路线变化
- 模块边界变化
- 验收标准变化

更新要求：

- 只记录可验证的内容。
- 不把临时脑暴写入正式主文档。
- 不在多个正式文档重复维护同一份“当前状态”。
- 顶层文档优先保持稳定，模块补充文档承接具体设计细节。
