# 2. 对 `Conversation与GEOCopilotChat层开发方案.md` 的判断

## 文档里正确的核心判断

这份方案最重要的判断是正确的：

```text
用户 URL 或上传页面
-> PageEvidencePack / PageContentProfile / RuleChecks
-> MethodSelector / StrategyPlanner
-> SafePromptPack / DeepSeekDiagnosis
-> ConversationSafePack
-> DeepSeek Copilot Turn
-> CopilotTurn validator
-> 保存对话回合与可追溯反馈
```

文档明确要求 Chat 层绑定 `analysis_id`、默认复用 snapshot、不重新抓取、不直接读取 raw HTML，并且个性化只来自用户显式提供的业务上下文。 这个边界非常关键，必须保留。

文档还明确：`ConversationSafePack` 是 Chat 层唯一模型输入，不允许直接拼接 raw HTML；`CopilotTurn Validator` 必须 fail closed；Chat 层不修改 `AnalysisResponse`。 这也是正确方向。

## 需要调整的地方

当前文档有几个地方需要收紧，否则实现时容易变复杂。

### 2.1 `ConversationSafePack` 不应默认塞完整 `DeepSeekDiagnosis`

文档里写 `diagnosis_summary: DeepSeekDiagnosis | None`，并说 token 压力变大后再改 compact summary。

我的判断：**v0 就应该设计成 compact summary，而不是先传完整 diagnosis。**

原因：

- `SafePromptPack` 已经包含 facts、rule checks、methods、strategy 和 evidence excerpts。
- `DeepSeekDiagnosis` 本身又包含 issues / actions / asset drafts / unknowns，容易重复。
- Chat 是多轮，每一轮都带完整 diagnosis 会迅速膨胀。
- 对话层更需要的是“当前页面问题摘要 + 可追溯 refs + 用户问题”，不是完整诊断对象。

建议改为：

```python
class DiagnosisCompactSummary(BaseModel):
    diagnosis_version: str
    geo_score: int | None = None
    top_issues: list[CompactIssue]
    top_actions: list[CompactAction]
    known_unknowns: list[str]
```

`ConversationSafePack` 中保留：

```python
diagnosis_summary: DiagnosisCompactSummary | None = None
```

并且 **diagnosis 不是 Chat 的强依赖**。没有生成 diagnosis 时，也可以基于 `SafePromptPack` 完成解释、优先级建议、证据请求等基础对话。

### 2.2 必须先修 `PageInputContext`，否则个性化是假功能

文档已经指出当前 `business_type` / `target_keywords` 不应继续只停留在 request model。 这是下一步第一优先级。

建议把它作为 Phase 1，而不是直接做 Chat。

正确实现顺序：

```text
AnalysisCreateRequest
-> PageInputContext
-> input_context.json
-> ConversationSafePack.input_context
```

同时要区分两类上下文：

| 类型 | 来源 | 作用 |
|---|---|---|
| `input_context` | 创建 analysis 时提交 | 页面级长期上下文 |
| `turn_user_context` | 每次追问临时提交 | 当前回合个性化覆盖 |

不要把用户每次聊天中的内容都静默写回 `input_context`。否则用户一句“假设我是 B2B SaaS”可能被永久当成真实业务背景。

### 2.3 上传页面要先重构 service，不要复制 URL 分析逻辑

当前 `PageEvidenceService.analyze()` 把 URL 校验、fetch、parse、profile、rule checks、methods、strategy、safe prompt、snapshot 保存全部写在一个方法里。

如果直接新增 `UploadedHtmlProvider`，很容易复制一遍主链路逻辑。更优方案是先抽出一个内部管道：

```python
def analyze_page_source(source: PageInputSource, language: str, input_context: PageInputContext) -> AnalysisResult:
    parsed = parse_html(source.html, source.base_url)
    ...
    save_snapshot(...)
```

URL 只是一个 source：

```python
FetchedUrlSource(
    input_url=url,
    final_url=fetched.fetch_info.final_url,
    html=fetched.html,
    fetch_info=fetched.fetch_info,
)
```

上传 HTML 也是一个 source：

```python
UploadedHtmlSource(
    declared_url=declared_url,
    filename=filename,
    sha256=upload_sha256,
    html=html,
)
```

这样才能保证 URL 和上传页面最终都产出同构 snapshot。

### 2.4 Conversation 存储要避免并发写坏文件

当前项目用 `SnapshotStorage` 保存 `raw.html`、`clean.md`、`evidence.json`、`rule_checks.json`、`safe_prompt_pack.json`、`analysis.json` 等文件。 继续用文件 snapshot 是合理的，但 Chat 会有连续写入。

建议 v0 文件结构改成：

```text
data/analyses/{analysis_id}/
  input_context.json
  conversations/
    default/
      conversation.json
      turns/
        000001_user.json
        000001_assistant.json
        000002_user.json
        000002_assistant.json
      summary.json
```

写入策略：

- 每个 turn 单独文件。
- `conversation.json` 只保存索引。
- 写文件使用临时文件 + atomic rename。
- v0 可以不做数据库，但必须保证一次模型失败不会污染有效历史。

### 2.5 DeepSeek settings 不要复制到 ConversationService

当前 `DiagnosisService` 内部定义了 `DeepSeekSettings`，并负责从 env 构建 client。 后续 Conversation 层也会调用 DeepSeek，如果再写一套 settings，会出现默认模型、timeout、retry 不一致。

建议把它移动到：

```text
apps/api/app/llm/settings.py
```

供 diagnosis 和 conversations 共用：

```python
class LLMSettings(BaseModel):
    provider: Literal["deepseek"]
    api_key: str
    base_url: str
    model: str
    timeout_seconds: float
    max_retries: int
    max_tokens: int
```

然后：

```text
DiagnosisService -> DeepSeekClient
ConversationService -> DeepSeekClient
```

共享同一个 provider 边界。

# 3. 成熟 GitHub 项目的借鉴取舍

你的文档列出的候选项目是合理的。GitHub 搜索确认这些项目都是真实、公开、非归档项目，例如 `assistant-ui/assistant-ui`、`CopilotKit/CopilotKit`、`vercel/chatbot`、`open-webui/open-webui`、`danny-avila/LibreChat` 都是 public 且未归档仓库。`assistant-ui/assistant-ui` `CopilotKit/CopilotKit` `vercel/chatbot` `open-webui/open-webui` `danny-avila/LibreChat`

## 最优取舍表

| 项目 | 是否接入 | 借鉴内容 | 不采用原因 |
|---|---:|---|---|
| `assistant-ui` | **v0 可接入** | React chat primitives、thread、composer、message rendering、custom runtime | 只作为 UI primitives，不让它决定后端协议 |
| `vercel/chatbot` | 不直接接入 | Next.js chat app 的状态、持久化、streaming 设计 | 依赖链和产品假设过重 |
| `CopilotKit` | v1 后再看 | generative UI、action、agent state | v0 不需要前端动作权限，过早接入会复杂化 |
| `LibreChat` | 不接入 | 多会话、provider、历史搜索能力清单 | 是完整聊天平台，不适合嵌入你的 GEO 主链路 |
| `Open WebUI` | 不接入 | 文件聊天、知识库、模型管理体验 | 产品形态太重，会吞掉 GEO 独立架构 |
| `Dify / AnythingLLM` | 不接入主链路 | 研究后台 / 内部知识库管理 | 会把确定性 MethodSelector 拉回不可控 RAG |

文档中“优先 assistant-ui、不 fork 大平台、不把 Dify / RAGFlow 放到正式用户请求路径”的判断应保留。

但要加一个工程保护：**assistant-ui 先通过安装、typecheck、最小页面验证，再正式采用。** 当前前端依赖只有 Next、React、React DOM、TypeScript。 页面也只是一个最小 URL 分析控制台。 所以前端集成风险不高，但仍要做兼容性 gate。
