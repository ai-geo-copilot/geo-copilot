# 2. 为什么不能直接用 RAGFlow / Dify / LlamaIndex 做核心

## 2.1 RAGFlow 很强，但太重，适合做研究后台

RAGFlow 官方定位是开源 RAG engine，强调复杂文档理解、模板化 chunking、可视化 chunk、引用溯源、多格式数据源和 re-ranking。它很适合处理论文、PDF、网页、表格、PPT 等研究资料。

但它自托管要求也不轻：CPU 至少 4 核、RAM 至少 16GB、磁盘至少 50GB、Docker 和 Docker Compose 等。

所以它适合：

```text
研究资料后台
论文问答
方法来源管理
人工整理 method cards
```

不适合：

```text
每次用户分析 URL 时都作为主链路检索系统
```

原因：你的主链路需要稳定、可测、可回归。RAGFlow 的价值是“资料召回”，不是“决定诊断策略”。

---

## 2.2 Dify 适合内部工具，不适合产品核心

Dify 的 Knowledge 功能适合导入数据、定义处理规则、管理 chunks、测试 retrieval、添加 metadata、调整 embedding 和 retrieval strategy。

它适合做：

```text
内部 GEO 助手
论文问答 bot
团队知识库
快速演示 demo
```

不适合做：

```text
你正式 API 的 MethodSelector
```

因为 Dify 的知识库输出仍然是面向“问答 / 应用增强”，而你的核心需要：

```text
RuleCheck.rule_id
failure_type
evidence_refs
page_type
readiness
→ 方法卡选择
→ 策略排序
→ 输出约束
```

这不是普通 RAG 能稳定解决的问题。

---

## 2.3 LlamaIndex + Qdrant 是后期增强，不是当前核心

LlamaIndex 的优势是数据连接、ingestion、index、query engine、agent、workflow 和复杂 context augmentation。  
Qdrant 的优势是 payload filter、布尔条件组合、全文匹配、向量检索等。

但你现在方法卡规模不会超过 20-30 个。这个规模下，上向量库反而会带来：

```text
依赖变多
测试变复杂
结果不稳定
调参成本增加
部署成本增加
错误归因困难
```

所以当前不接 Qdrant。  
等你的方法卡超过 100 个、出现行业包、需要语义召回和 metadata filter 时，再接：

```text
LlamaIndex + Qdrant
```

# 5. Method Pack 应该怎么设计

## 不要只写方法卡

你需要三类对象：

```text
MethodChunk
RuleMethodBinding
StrategyGroup
```

---

## 5.1 MethodChunk

```json
{
  "method_ref": "chunk_geo_definition_unit_001",
  "title": "Definition Unit Construction",
  "priority": "P0",
  "strategy_group": "absorption_foundation",
  "applies_to_page_types": ["article", "product", "docs", "comparison", "landing"],
  "applies_to_rule_ids": [
    "content.definition_unit_missing",
    "absorption.readiness_low"
  ],
  "applies_to_failure_types": [
    "absorption_blocker"
  ],
  "uses_profile_fields": [
    "page_type",
    "primary_entity_candidates",
    "answer_units",
    "absorption_readiness"
  ],
  "text": "Add a concise, visible definition or summary unit near the top of the page so generative engines can identify the page's primary entity and reuse the answer-ready content.",
  "action_steps": [
    "Identify the primary entity from PageContentProfile.",
    "Check whether a definition-like answer unit already exists.",
    "Create a visible definition block near the top of the main content.",
    "Use only facts already present in PageEvidencePack.",
    "Bind the recommendation to evidence_refs."
  ],
  "expected_artifacts": [
    "top_summary_block",
    "definition_block"
  ],
  "guardrails": [
    "Do not invent product capabilities.",
    "Do not create unsupported factual claims.",
    "If the primary entity is unclear, pair with primary entity clarification first."
  ]
}
```

---

## 5.2 RuleMethodBinding

不要只靠 method 里写 `applies_to_rule_ids`。  
单独建映射表，更容易维护。

```json
{
  "rule_id": "content.definition_unit_missing",
  "failure_type": "absorption_blocker",
  "default_methods": [
    "chunk_geo_definition_unit_001",
    "chunk_geo_primary_entity_001"
  ],
  "required_strategy_group": "absorption_foundation",
  "severity_override": {
    "failed": "P0",
    "warning": "P1"
  }
}
```

这样后期规则变多时，不需要改每张方法卡。

---

## 5.3 StrategyGroup

```json
{
  "strategy_group": "claim_evidence_strengthening",
  "rank": 40,
  "description": "Improve verifiability and citation readiness by binding claims, statistics, and source cues.",
  "must_run_after": [
    "critical_safety",
    "selection_foundation",
    "absorption_foundation"
  ]
}
```

这样最终输出不是“方法列表”，而是“修复顺序”。

# 9. 什么时候才接向量库

我给一个硬标准。

满足下面任意两个，再接 Qdrant：

```text
1. MethodChunk 超过 100 个
2. 需要行业方法包，例如 SaaS / e-commerce / B2B / docs / local service
3. 一个 failure_type 对应 10 个以上候选方法
4. 需要按 source_ref / evidence_level / paper / page_type 做复杂过滤
5. 需要从论文 chunk 中动态召回 supporting rationale
```

到那时架构升级为：

```text
MethodSelector deterministic candidates
        ↓
Qdrant metadata filter + vector retrieval
        ↓
rerank
        ↓
StrategyPlanner
```

注意顺序：**先规则候选，再向量增强**。  
不要让向量库从零开始决定方法。

排序：


```text

```

不改：

```text

```

---

```text

```


